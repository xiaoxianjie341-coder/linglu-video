# `+` 图片入口设计

**日期：** 2026-04-24  
**状态：** 已确认方向，待进入实施计划

## 目标

把当前输入框左侧还处于禁用状态的 `+` 按钮，升级为真正可用的图片入口，并明确它在图片模式与视频模式里的默认语义。

这次设计的目标不是单纯“支持上传图片”，而是建立一个清晰、一致、可扩展的用户心智：

- 用户看到 `+`，就知道这里可以“放图片”
- 用户点击 `+` 后，不需要先学习复杂概念
- 图片上传完成后，系统必须把这张图当前扮演的角色说清楚

## 设计原则

- `+` 是一级入口，不是隐藏设置项
- 入口动作统一，模式语义分流
- 第一点击尽量短，不先弹复杂菜单
- 图片模式和视频模式不强行共用同一套解释
- V1 先把主路径做清楚，不抢跑大而全附件中心

## 非目标

- 不把链接抓取、网页解析、素材库、模板市场全部塞进 `+`
- 不在 V1 就做完整“附件中心”或复杂抽屉
- 不在 V1 就做图片模式的主体/风格/构图角色选择器
- 不在 V1 就做视频模式的尾帧、多关键帧、镜头级时间轴编辑
- 不单独做一条脱离现有文本生成链路的全新“图生视频工作流”

## 当前基线

当前产品里：

- `components/input-panel.tsx` 左侧 `+` 按钮仍是禁用占位
- 图片模式只支持文本输入，不支持从输入区放图
- 视频模式的主流程仍然是文本或链接 -> 分镜 -> 视频
- 视频生成底层已经具备“从参考图生成视频”的能力，但当前主链路仍然以生成出来的 `3x3` 总分镜作为视频预览参考图，还没有把用户主动上传的图片接入这条链路

这意味着本次设计不能只做一个表面上传控件，而是要明确：

1. `+` 点下去做什么
2. 上传后的图片在两种模式里分别进入哪里
3. 上传后的图片如何进入现有 request / run / pipeline

## V1 总体定义

### 1. `+` 的职责

`+` 在 V1 中只承担一个动作：

**点击后直接打开系统文件选择器，允许用户放入图片。**

这里不先弹菜单，不先让用户做“图片 / 链接 / 素材库”的选择。

### 2. 统一入口，不统一语义

虽然动作统一，但上传后的图片语义由当前模式决定：

- `图片模式`：默认语义是 `参考图`
- `视频模式`：默认语义是 `起始帧`

所以 V1 复用的是“导入图片”这个动作，不复用“这张图到底代表什么”的解释。

## 图片模式

### 默认语义

图片模式中，用户通过 `+` 上传的图片默认是：

**参考图**

它的职责是帮助系统约束接下来要生成的静态图片，而不是直接把原图当结果输出。

### 界面表现

上传后，图片应进入输入框下方的轻量缩略图区：

- 每张图显示缩略图
- 每张图旁显示统一标签：`参考图`
- 支持删除单张图片
- 支持继续追加上传，直到达到 V1 上限

### V1 能力边界

图片模式第一版不细分：

- 主体参考
- 风格参考
- 构图参考

这些概念可以在未来演进，但 V1 只对用户暴露一个统一、易懂的词：

**参考图**

## 视频模式

### 默认语义

视频模式中，用户通过 `+` 上传的第一张图片默认是：

**起始帧**

它不是普通参考图，也不是素材附件，而是视频开场画面的视觉锚点。

### 界面表现

视频模式不使用图片模式那种缩略图条，而是使用更强语义的单卡片位：

- 上传后显示一张更醒目的图片卡
- 卡片标题或标签明确写 `起始帧`
- 支持删除并重新上传
- 空状态文案明确提示“上传一张图，作为视频的起始帧”

### 与当前链路的关系

因为现有视频主流程仍然是：

`文本 / 链接 -> 分镜规划 -> 视频生成`

所以 V1 里的 `起始帧` 不是一条完全独立的新视频模式，而是：

**现有视频流程中的开场视觉锚点**

也就是说：

- 用户仍然需要文本输入来描述内容、动作或剧情
- `起始帧` 负责给分镜生成链路一个更明确的开场视觉起点
- V1 不承诺完整的多关键帧视频编辑能力

更具体地说，V1 的 `起始帧` 先作用在**分镜生成与开场画面约束**上，而不是直接把现有总分镜参考图替换掉。这样既保持“视频模式里这张图是起始帧”的用户心智，也不强行推翻当前的 `grid_preview` 主流程。

## 适用范围

这次 `+` 不是只给首页大卡片补一个上传按钮，而是 `InputPanel` 体系下的统一入口。

因此 V1 适用范围包括：

- 首页主输入区
- run 详情页里的“新建创作”输入区
- 小屏与大屏都可见，不做“桌面端才有 `+`”的隐藏版本

如果某个场景暂时无法支持，必须在实施前明确缩小范围，而不是在实现时默默漏掉。

## 模式切换规则

由于图片模式和视频模式的图片语义不同，V1 不应该把同一组上传图片自动重解释。

因此默认规则定为：

- 两种模式各自维护独立的图片草稿状态
- 在图片模式上传的 `参考图`，不会在切到视频模式后自动变成 `起始帧`
- 在视频模式上传的 `起始帧`，也不会在切回图片模式后自动变成 `参考图`

这样做是为了避免用户对图片角色产生误解。

## 文案策略

### 入口层

入口层文案尽量统一、短促：

- `+`
- 悬停提示：`上传图片`

### 上传后

一旦图片进入当前模式，文案必须开始显式区分语义：

- 图片模式：`参考图`
- 视频模式：`起始帧`

不要让用户只看到一张图片缩略图，却不知道系统会怎么使用它。

### 空状态示例

- 图片模式：`上传图片，作为这次出图的参考图`
- 视频模式：`上传图片，作为视频的起始帧`

## V1 约束

为了避免入口失控，V1 增加以下约束：

- 只支持图片文件，不支持把链接导入塞进 `+`
- 图片模式支持多张 `参考图`
- 视频模式第一版只支持单张 `起始帧`
- 视频模式先不支持 `尾帧`
- 视频模式先不支持多关键帧
- 上传结果都必须能被回显到任务输入卡中，方便用户复核

## 未来演进顺序

如果这条入口验证成立，后续优先级建议如下：

1. 视频模式补 `尾帧`
2. 图片模式补更细的参考角色
3. 增加 `粘贴图片`
4. 增加“最近使用图片”
5. 最后才考虑把链接或其他素材输入并入 `+`

也就是说，未来扩展优先沿着“更强图片输入能力”走，而不是把 `+` 立刻做成综合附件中心。

## 设计结论

这次 `+` 的 V1 设计可以压成一句话：

**`+` 在交互上是一个直通的放图入口，但在产品语义上按模式分流：图片模式里是参考图，视频模式里是起始帧。**

这条路线同时满足三件事：

- 保留“点一下就能放图”的直觉
- 向主流产品看齐，不强行统一图片与视频的语义
- 不把第一版做成复杂、沉重的附件系统
