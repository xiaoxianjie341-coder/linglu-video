# 即梦参考式产品壳重构设计

**日期：** 2026-04-17
**状态：** 已确认方向，待进入实施计划

## 目标

把当前偏内部工具感的 Web 界面，重构成更像对外产品的统一前端壳子。整体框架参考即梦首页的成熟产品感，但页面内容必须严格围绕现有真实能力展开，不新增假的生态入口、空按钮或未实现功能。

## 设计原则

- 参考即梦的轻量产品框架，而不是照搬它的业务生态
- 首页负责激发创作，不承载完整结果
- 生成后跳入第二页，第二页承担任务执行与结果展示
- 历史页和设置页共享统一产品壳，弱化“内部控制台”气质
- 只暴露真实能力：输入一句话或链接，生成故事提要、`3x3` 总分镜和最终视频

## 非目标

- 不做发现流、活动流、模板市场、社区、API 商店等生态模块
- 不伪造图片生成、画布、智能体市场、多模态工作区等当前不存在的能力
- 不做完整聊天产品，只借“任务流/对话感”的节奏组织生成页面

## 整体结构

产品改成统一壳下的四个核心路由：

1. `/` 首页：创作入口页
2. `/runs/[id]` 任务页：生成中的执行现场与结果页
3. `/history` 历史页：作品记录
4. `/settings` 设置页：接入配置

左侧导航第一版只保留三个入口：

- 创作
- 历史
- 设置

## 首页 `/`

### 角色

首页是“开始创作”的地方，目标是让用户第一眼就知道：输入一句话，就能生成故事视频。

### 页面结构

- 左侧窄导航
- 中央主舞台
- 上半屏为主视觉与大输入卡
- 下半屏为轻量能力说明和最近任务入口

### 核心内容

#### 1. 主视觉区

- 一句产品标题，例如“输入一句话，生成故事视频”
- 一段简短说明，解释当前工作流：扩成故事提要、生成 `3x3` 总分镜、生成视频

#### 2. 大输入卡

输入卡参考即梦截图的中心大框结构，但字段只保留真实能力：

- 文本 / 链接输入
- 品牌调性
- 视频引擎
- 视频模型
- 视频时长
- 发送 / 生成按钮

参数栏收在输入卡底部或下边缘，避免像当前版本那样像表单面板。

### 首页可用能力边界

第一版首页只展示**已接入且可执行**的生成选项：

- 默认只展示当前真实可用的视频引擎和模型
- `Kling`、`即梦` 虽然在设置页中保留“预接入”状态，但在首页不展示为可选生成项
- 不允许出现“可以选择但实际跑不通”的 provider

### 首页可用性判定来源

首页的“可不可生成”不按前端本地表单猜测，而按服务端的**真实运行时可用性**判断：

- 文本规划是否可用：按第一版真实可执行的 planner 运行时判断
- 总分镜图是否可用：固定要求 `OpenAI` 媒体运行时可用
- 视频生成是否可用：要求至少存在一个**已实现且运行时可用**的视频 provider

第一版的产品规则进一步收紧为：

- 首页创作主流程按 **OpenAI 全链路** 作为唯一真实可执行路径
- `Linglu` 配置仍保留在设置页，但第一版不作为首页创作路径的可用判断来源
- 首页默认只暴露 `OpenAI` 视频生成
- 即使设置页里保存了 `Kling` 或 `即梦` 配置，只要 provider 仍未实现，就不在首页展示
- 首页必须消费一个服务端“有效可用性”结果，而不是单纯读取“是否保存过 key”

### 首页可用性接口契约

第一版必须新增或扩展一个服务端 preflight 契约，至少返回：

- `plannerReady`
- `storyboardImageReady`
- `availableVideoProviders`
- `canGenerate`
- `blockingReason`

并且：

- 首页禁用逻辑必须消费这套 preflight 结果
- 提交接口也必须复用同一套 preflight 校验
- 不允许首页显示可生成、提交后才因同一原因立刻失败

### 首页前置条件与降级

首页必须处理未配置状态：

- 如果当前没有可用的媒体生成配置，则禁用“生成”按钮
- 如果当前没有可用的文本规划配置，也禁用“生成”按钮
- 输入卡内展示明确提示，引导用户前往设置页完成配置
- 如果配置失效，也在首页内直接提示，不等到提交后才报错

这里的“未配置 / 配置失效”同样以服务端运行时判断为准，包含：

- 已保存设置但缺失环境变量补充时的真实可用性
- 已保存配置但 provider 尚未实现时的不可用状态
- OpenAI 主创作链路不可执行时的阻断状态

#### 3. 能力说明卡

替代即梦首页的一排生态入口，改为我们自己的真实能力卡：

- 一句话扩成故事提要
- 自动生成 `3x3` 总分镜
- 基于总分镜生成视频
- 支持文本或公开链接

#### 4. 最近任务入口

首页底部保留一个轻量“最近任务”区：

- 仅展示少量最近任务卡片
- 用于快速回到上一次创作
- 不做内容发现流

### 行为

用户点击“生成”后，不在首页内展开长结果，而是跳转到 `/runs/[id]`。

## 任务页 `/runs/[id]`

### 角色

任务页是“系统正在替你执行生成任务”的页面。它要有类似对话式任务流的节奏，但不是一个真正的聊天产品。

### 页面结构

- 顶部仍使用统一壳和左导航
- 主区域为单列时间线 / 对话式任务流
- 内容按“输入 -> 处理中 -> 产物插入”的顺序往下展开

### 内容组织

#### 1. 用户输入回显

页面顶部先展示用户刚刚提交的原始内容，像一条“用户消息”。

#### 2. 系统执行流

按阶段显示生成状态卡：

- 排队中
- 理解内容
- 规划故事提要与分镜
- 生成 `3x3` 总分镜
- 生成最终视频

每个阶段都要支持：

- 待开始
- 处理中
- 成功
- 失败
- 上游失败导致未开始

#### 3. 产物插入

当阶段完成时，直接把结果插入时间线：

- 故事提要卡
- `3x3` 总分镜
- 最终视频

错误不要只挂在页面顶部，而要落在对应阶段卡片里。

### 内容边界

第一版任务页显式展示的是“故事提要”，不是一篇完整长剧本：

- 来源于规划阶段的 `content_summary`
- 作用是让用户看到系统如何把一句话扩成一个更清晰的故事方向
- 不额外承诺完整分场剧本或长篇文案编辑能力

### 阶段状态承载方式

第一版任务页要支持稳定的阶段展示，因此必须补最小数据契约，而不是只靠现有 `status / phaseLabel / error` 猜测：

- `activePhase`：当前或最近一次执行到的阶段
- `failedPhase`：失败发生的阶段

前端任务流卡片按以下数据生成：

- 当前阶段由 `activePhase` 决定
- 已完成阶段按 `status + activePhase` 派生为成功
- 失败阶段由 `failedPhase` 决定
- 失败后的后续阶段标记为“上游失败导致未开始”

也就是说，第一版允许保留现有全局 `status`，但必须新增足够的阶段级持久化字段，保证刷新和历史回看时不丢失失败位置。

### 产物增量写回

任务页要求边执行边插入产物，因此第一版必须把以下结果改成**增量写回**：

- 规划完成后立即写回故事提要
- 总分镜图完成后立即写回 storyboard 产物
- 视频完成后立即写回最终视频

不能继续沿用“全部完成后一次性写入最终结果”的方式，否则任务页设计无法落地。

### 关键取舍

- 不伪装成聊天助手，不做多轮对话输入
- 只借“连续任务流”和“消息式节奏感”
- 用任务卡代替当前多个并列大面板

## 历史页 `/history`

### 角色

历史页是“创作记录页”，不是 runId 文本日志列表。

### 页面结构

- 统一产品壳
- 顶部标题区
- 下方卡片列表

### 每条卡片显示

- 原始输入摘要
- 当前状态：生成中 / 已完成 / 失败
- 缩略图：优先视频缩略图，其次总分镜
- 元信息：创建时间、视频引擎、时长
- 主入口：继续查看

### 失败态

失败任务卡片保留：

- 失败状态
- 失败原因摘要
- 重新进入任务页的入口

## 设置页 `/settings`

### 角色

设置页是“接入配置页”，不是内部调试台。

### 页面结构

拆成独立配置卡，而不是一整页长表单：

- OpenAI 媒体底座
- Kling 预接入
- 即梦预接入
- 文本规划（灵鹿）

### 表达原则

每张配置卡都要说明：

- 这个配置用于什么
- 当前是否已配置
- 下一步该怎么做

### 约束

- 保留当前真实配置能力
- 对预留但未实现的 provider，明确标记“已预留接入”
- 不制造“已经支持”的错觉

## 视觉方向

采用混合方案：

- 首页高拟合参考即梦截图的框架感
- 任务页、历史页、设置页更多按我们的业务重排

视觉特征包括：

- 浅色、轻留白、产品感强的外壳
- 左侧窄导航
- 中间大面积内容舞台
- 输入区作为页面主角
- 结果区采用更模块化、更具节奏感的卡片组织

## 影响范围

- `app/layout.tsx`
- `components/studio-page.tsx`
- `components/input-panel.tsx`
- `components/progress-panel.tsx`
- `components/storyboard-grid.tsx`
- `components/video-result.tsx`
- `components/history-list.tsx`
- `components/settings-form.tsx`
- 必须新增任务详情页路由：`app/runs/[id]/page.tsx`
- 必须抽出共享产品壳组件：侧导航、页面头部、任务流卡片等
- 必须新增首页“有效可用性”数据来源（可复用或扩展现有设置接口，也可新增专用接口）
- 必须让提交接口复用首页同一套 preflight 校验
- 必须扩展 run 数据契约以持久化 `activePhase` / `failedPhase`
- 必须把故事提要、总分镜、视频结果改成增量写回
- 必须新增任务页阶段映射层，把持久化阶段状态映射成时间线卡片状态

## 验证标准

- 首页第一眼是“创作入口”，而不是后台工作台
- 点击生成后进入任务页，而不是继续停留在首页堆叠结果
- 历史页表现为作品记录，而不是纯日志
- 设置页表达为接入配置，而不是内部调试面板
- 首页不会暴露未接入、不可运行的 provider 选项
- 未配置状态在首页即可被识别并阻断，不会等提交后才失败
- 整体产品壳借鉴即梦截图的成熟框架感，但页面中不存在当前业务没有的假功能
